- receive_pack: the `report-status` capability is unconditionally advertised in `_build_receive_pack_advertisement` and the status report (`unpack ok` + per-ref `ok`/`ng` lines) is always sent in the response, regardless of whether the client requested it. Per the git smart-HTTP spec, the server should only send the report-status body if the client included `report-status` in its capability list. The client's capabilities arrive NUL-separated on the first command line (e.g. `<old> <new> <ref>\0report-status\n`); currently `receive_pack` in `git_smart_http_views.py` strips everything after the NUL and discards it. Fix: parse the capability string from the first command line, store it in a set, and gate the status-report response body on `"report-status" in client_capabilities`. If the client did not request `report-status`, the response body should be empty (just the flush packet, or nothing — check gitprotocol-pack[5] for exact behavior).
