- Consider adding more ruff lint rules: `N` (pep8-naming), `C4` (flake8-comprehensions)
- receive_pack: the `report-status` capability is unconditionally advertised in `_build_receive_pack_advertisement` and the status report (`unpack ok` + per-ref `ok`/`ng` lines) is always sent in the response, regardless of whether the client requested it. Per the git smart-HTTP spec, the server should only send the report-status body if the client included `report-status` in its capability list. The client's capabilities arrive NUL-separated on the first command line (e.g. `<old> <new> <ref>\0report-status\n`); currently `receive_pack` in `git_smart_http_views.py` strips everything after the NUL and discards it. Fix: parse the capability string from the first command line, store it in a set, and gate the status-report response body on `"report-status" in client_capabilities`. If the client did not request `report-status`, the response body should be empty (just the flush packet, or nothing — check gitprotocol-pack[5] for exact behavior).
- upload_pack: implement proper want/have negotiation so we send ACK and a minimal thin pack instead of always NAK + full pack
- git.py `parse_commit`: the header parser splits on the first `\n\n` to separate headers from the message body, but GPG-signed commits embed a `gpgsig` header whose value contains blank lines, which causes the split to occur inside the signature rather than at the true header/body boundary. Fix: parse headers line-by-line, accumulating continuation lines (lines starting with a space are folded continuations per the git object format), and only treat a truly blank line as the end of the header section.
- git.py `TreeEntry.type`: symlinks (mode `120000`) are correctly returned as `"blob"`, but submodule/gitlink entries (mode `160000`) are also returned as `"blob"` when they should be `"commit"`. Submodule support will require broader work: the model and code-browsing views will need to handle the `"commit"` entry type, resolve the submodule URL from `.gitmodules`, and decide how to link or display the pinned commit.
- receive_pack: only `refs/heads/` and `refs/tags/` namespaces are supported; pushes to other namespaces (e.g. `refs/notes/`, `refs/replace/`) are rejected with `ng <refname> unsupported ref`. Add proper support for arbitrary ref namespaces — likely requires a generic ref type in the model or storing the full ref name rather than splitting into type + short name.
